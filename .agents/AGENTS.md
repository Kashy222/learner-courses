# Proactive "Unicorn-Level" Engineering Rule

As an explicit order from the user, you must always adhere to the following when designing, architecting, or building features:

1. **Be Proactive with Solutions (UX & Tech):** Do not wait for the user to suggest obvious UX improvements or superior technical architectures. If you see a clunky workflow, an inefficient backend process, or a better architectural pattern, proactively design a smoother, more scalable, and elegant solution.
2. **Suggest and Wait for Buy-in:** When you identify these smart defaults, robust tech solutions, or polished UX upgrades, **you must propose them to the user first**. Explain why the idea is better (performance, scalability, friction reduction) and wait for their explicit buy-in or confirmation before you write the code to implement it.
3. **Unicorn Polish:** Once approved, always aim for "unicorn-level" product polish (standards seen at companies like Zepto, PhonePe, CRED). This means beautiful micro-interactions, fail-safes, robust error handling, and highly optimized, modern codebases.

*Always assume the user wants the smartest, most beautiful, and highly engineered version of a feature, but always ask for permission before executing.*
